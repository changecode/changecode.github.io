分布式事务DTS

分布式事务服务(Distributed Transaction Service DTS)是一种分布式事务框架，用来保证在分布式微服务环境下
端到端业务操作的最终一致性

常用的分布式技术

1、本地消息表

核心思想在于分布式系统在出来任务时通过消息日志的方式来异步执行。消息日志可以查查至本地文本、数据库
或消息队列，然后再通过业务规则通过定时任务或者人工自动重试。 以转账为例

第一步，对用户id为A的账户扣款1000元，通过本地事务将事务消息(包括本地事务id、支付账户、收款账户、金额
、状态等)插入至消息表. 伪代码

Begin transaction
update user_accout set amout = amout - 1000 where userId = ‘A’
insert into trans_message(xid,payAccout,recAccout,amout,status) values(uuid(),’A’,’B’,1000,1);
end transaction
commit;

第二步，通知对方用户id为B，增加1000元，通常通过消息MQ的方式发送异步消息，对方订阅并监听消息后自动
触发转账的操作；为了保证幂等性，防止触发重复的转账操作，需要在执行转账操作方新增一个trans_recv_log
表来做幂等，在第二阶段收到消息后，通过判断trans_recv_log表来检测相关记录是否被执行，如果未被执行则
会对B账户余额执行加1000元的操作，并会将记录增加至trans_recv_log，事件结束后通过回调更新trans_message
的状态值

2、消息中间件

2-1 非事务消息中间件

仍以转账为例，很难保证在扣款完成之后对MQ投递消息的操作就一定能成功，这样一致性很难保证。伪代码

try{
boolean result = dao.update(model);
if(result) {
mq.send(model);
}
}catch(Exception ex) {
rollback();
}
上述代码可能会出现如下几种情况:
1、操作数据库成功，向MQ投递消息也成功，属于正常情况
2、操作数据库失败，不会向MQ投递消息
3、操作数据库成功，但向MQ中投递消息失败，向外抛出异常，更新数据库操作回滚

从上面分析的几种情况来看，基本上能确保，发送消息的可靠性。然后在可靠消费者端的问题

1、接收者取出消息后，消费者对应的业务操作要执行成功。如果业务执行失败，消息不能失效或者丢失
需要保证消息与业务操作一致

2、尽量保证消息的幂等性。如果出现重复消息投递，能够进行幂等而不对业务产生影响

2-2 支持事务的消息中间件

RocketMQ

第一阶段，RocketMQ在执行本地事务之前，会先发送一个Prepared消息，并且会持有这个消息的接口回查
地址

第二阶段，执行本地事务操作

第三阶段，确认消息发送，通过第一阶段拿到接口地址URL执行回查，并修改状态，如果本地事务成功，则
修改状态为已提交，否则修改状态为已回滚

其中，如果第三阶段的确认消息发送失败后，RocketMQ会有定时任务扫描集群中的事务消息，如果发现
还处于prepare状态的消息，它会向消息发送者确认本地四五是否已执行成功。RocketMQ会根据发送端
设置的策略来决定是毁滚还是继续发送确认消息，这就保证了消息的发送与本地事务同时成功或同时
失败。 再回到上面的例子，如果用户A的账户余额已经减少，且消息已经发送成功，作为消费者用户B
开始消费这条消息，这个时候就会消费失败和消费超时两个问题，解决超时问题的思路就是一种重试，
直到消费端消费成功，整个过程中有可能会出现消息重复的问题，就需要采用幂等方案来进行处理

3、2PC协议(两阶段)

当一个全链路的端到端业务操作，常常会跨多个节点、多个应用、为了能够保证全局事务的ACID特性，
需要引入一个协调组件来控制所有服务参与者的操作结果，根据所有参与者的反馈结果来决定整个
分布式事务是提交还是回滚的结果。

第一阶级： 准备阶段。事务协调者向各个服务应用发送prepare请求，服务应用在得到请求后做预处理
操作，预处理可能是做预检查，也可能把请求临时存储，可以理解为是一种试探性提交。步骤如下：

1、事务协调者会问所有的参与服务，是否可以提交操作
2、各个参与者开始事务执行的准备工作: 如资源上锁，预留资源，写回滚/重试的log
3、参与者响应协调者，如果事务准备工作成功，则回应可以提交，否则回应拒绝提交

第二阶段： 提交/回滚阶段。 事务真正提交或者回滚的阶段。 如果事务协调者发现事务参与者有一个在
prepare阶段出现失败，则会要求所有的参与者进行回滚。如果协调者发现所有的参与者都prepare操作都是
成功的，那么将向所有的参与者发出提交请求，这时所有参与者才会正式提交。由此保证要不全部成功，要不
全部失败。 步骤如下：

1、如果所有的参与者都回应可以提交，那么协调者向所有参与者发送正式提交的命令。 参与者完成正式提交
并释放资源，然后回应完成，协调者收集各个服务的完成回应后结束事务

2、如果有一个参与者回应拒绝提交，那么协调者向所有的参与者发送回滚操作，并释放所有的资源，然后回应
回滚完成，协调者收集各个服务应用的回滚返回后，取消整体的分布式事务

缺点：

1、单点问题： 协调者在整个二阶段中的作用非常重要，一旦服务不可用，影响整个分布式系统运行

2、同步阻塞： 二阶段提交执行过程中，所有服务参与者需要服从协调者的统一调度
